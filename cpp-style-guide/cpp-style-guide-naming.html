1. Правила именованияFalse

1.1. Используйте правильный английский для идентификаторовFalse

Все имена и идентификаторы пишутся на английском (без использования транслитерации и т.д.)

ПРАВИЛЬНО 
file_name
НЕПРАВИЛЬНО 
imya_fayla

1.2. Избегайте “бессодержательных” имен и нестандартных сокращенийFalse

1.3. Сокращения (аббревиатуры) подчиняются общим правиламFalse

Если в качестве имен (части имен) используются сокращения, аббревиатуры и акронимы, то последние должны подчиняться общим правилам.

ПРАВИЛЬНО 
ExportHtmlSource();

OpenDvdPlayer();
НЕПРАВИЛЬНО 
ExportHTMLSource();

OpenDVDPlayer();

1.4. Именование папок и файлов в нижнем регистре через подчеркиваниеFalse

Имена файлов и папок исходного кода пишутся в нижнем регистре через подчеркивания. Исключения в части регистра допустимы для библиотек сторонних разработчиков и при наличии технологических ограничений (например, файлы системы сборки CMake вида CMakeList.txt).

При выборе имен следует избегать повторения имен стандартной библиотеки и распространных библиотек сторонних разработчиков. Приветствуется использование достаточно длинных и информативных имен.

ПРАВИЛЬНО

report_generator

report_generator.h
НЕПРАВИЛЬНО

db.h

crepgen.hpp

1.5. Используйте стандартные расширения для файлов с исходными кодамиFalse

Используйте для различных типов файлов следующие расширения:

  • «h» - заголовочные файлы С++
  • «cpp» - файлы реализаций С++
ПРАВИЛЬНО

report_generator.h

report_generator.cpp
НЕПРАВИЛЬНО

crepgen.H

crepgen.hpp

report_generator.cxx

report.c

1.6. Пишите имена типов в смешанном регистре с заглавной буквы (без подчеркиваний)False

Имена всех типов (классов, структур, псевдонимов типа, перечислений и параметров шаблонов) пишите в смешанном регистре с заглавной буквы без подчеркиваний. Каждое новое слово в имени типа начинайте с заглавной буквы.

ПРАВИЛЬНО 
// classes and structs
class Rational { ...

class UrlTableTester { ...

struct UrlTableProperties { ...

// using aliases
using PropertiesMap = hash_map<UrlTableProperties*, std::string>;

// enums
enum UrlTableErrors { ...
НЕПРАВИЛЬНО 
class rational { ...

1.7. Пишите имена переменных в нижнем регистре через подчеркиванияFalse

Имена всех переменных (локальных, глобальных, параметров функций и полей) пишите в нижнем регистре с использованием символа подчеркивания для разделения отедельных слов.

ПРАВИЛЬНО 
str_message
НЕПРАВИЛЬНО 
strmessage

str_message

StrMessage

Имена полей структур подчиняются общему правилу именоваания переменных. Имена полей классов завершайте символом подчеркивания.

ПРАВИЛЬНО 
class FileInfo {
  // ...
private:
  std::string file_path_;
};

struct ComplexNumber {
public:
  double re;
  double im;
  // ...
};
НЕПРАВИЛЬНО 
class FileInfo {
  // ...
private:
  std::string m_file_path_; // нестандартный префикс
  std::string file_alias;   // нет стандартного суффикса
};

1.8. Пишите имена неизменяемых и статических переменных в смешанном регистре с префиксом ‘k’False

Используйте префикс ‘k’ для имен неименяемых (const, constexpr), глобальных и статических (static) переменных. Если смешанный регистр затрудняет чтение, то можно использовать подчеркивание для отделения префикса.

ПРАВИЛЬНО 
const int kDaysInAWeek(7);
const int kAndroid8_0_0(24); // Android 8.0.0

static int kCallCounter(0);
НЕПРАВИЛЬНО 
const int k_days_in_a_week(7);

1.9. Используйте стиль именования константых переменных для значений перечисленийFalse

Пишите имена значений перечислений аналогично константным переменным, т.е. в смешанном регистре с префиксом ‘k’.

ПРАВИЛЬНО 
enum UrlTableErrors {
  kSuccess = 0,
  kErrorOutOfMemory,
  kErrorMalformedInput,
};
НЕПРАВИЛЬНО 
MaxIterations
max_iterations
MAXITERATIONS

1.10. Пишите имена функций в смешанном регистре начиная с символа верхнего регистраFalse

Имена функций пишите в смешанном регистре начиная с символа верхнего регистра. Выбирайте имена функциям в зависимости от их семантики. Функциям (возвращают некоторое значение) давайте имена в соответствии с тем, что они возвращают. Процедурам (ничего не возвращают) давайте имена в соответствии с тем, что они делают. Рекомендуется начинать имя функции с глагола (английского).

ПРАВИЛЬНО 
AddTableEntry()
DeleteUrl()
OpenFileOrDie()
GetName()
ComputeTotalWidth()
НЕПРАВИЛЬНО 
GetName()
widthCompute()

Для парных функций используйте парные имена.

  • get/set
  • add/remove
  • create/destroy
  • start/stop
  • insert/delete
  • increment/decrement
  • old/new
  • begin/end
  • first/last
  • up/down
  • min/max
  • next/previous
  • old/new
  • open/close
  • show/hide
  • suspend/resume
  • etc.
ПРАВИЛЬНО 
bool AddPage() {
    ...
}

bool RemovePage() {
    ...
}
НЕПРАВИЛЬНО 
bool AddPage() {
    ...
}

bool DestroyPage() {
    ...
}

1.12. Использование распространенных сокращений и соглашений при именованииFalse

Используйте множественные формы существительных для обозначения коллекций объектов.

ПРИМЕР:
vector<Point> points;

Используйте префикс “i” для переменных, определяющих порядковый номер некоторого объекта.

ПРИМЕР:
i_point

Например, такой префикс можно использовать для именования индексов строк (row) и столбцов (column) при написании циклов обхода матрицы.

ПРАВИЛЬНО 
for (ptrdiff_t i_row(0); i_row < m.RowCount(); i_row += 1) {
  for (ptrdiff_t i_col(0); i_col < m.ColumnCount(); i_col += 1) {
      // ...
  }
}
НЕПРАВИЛЬНО 
for (int i(0); i < m.RowCount(); i += 1) {
  for (int j(0); j < m.ColumnCount(); j += 1) {
      // ...
  }
}

Используйте префиксы “is”, “has”, “can”, “should” для имен переменных и функций, определяющих логические свойства некоторого объекта.

ПРИМЕР:
bool is_book(true);

bool CanMove() const;

Для переменных, определяющих число некоторых объектов, можно использовать префикс “n” (сокращение от “number_of”).

ПРИМЕР:
n_point = points.size();

Пишите макроопределения в верхнем регистре с использованием подчеркивания в качестве разделителя.

ПРИМЕР:

1.14. Используйте одиночные заглавные буквы для идентификаторов типизированных параметров шаблоновFalse

ПРИМЕР:
template<class T>

template<typename C, typename D>

1.15. Избегайте отрицаний в идентификаторахFalse

Используйте имена переменных без отрицаний. Наличие отрицания в имени переменной сильно усложняет понимание логических выражений, особенно, если в последних присутствует логическое отрицание.

ПРАВИЛЬНО 
bool is_error(false);
bool can_move(true);

if (!is_error) {
    ...
}
НЕПРАВИЛЬНО 
bool is_not_error(true);
bool cant_move(false);

if (!is_not_error) {
    ...
}

1.16. Избегайте запрещенных идентификаторовFalse

Запрещено начинать имена с подчеркивания (резервируется для реализации компиляторов).

ПРАВИЛЬНО 
file_name
НЕПРАВИЛЬНО 
_fileName
